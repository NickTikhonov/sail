# sail

sail treats a TypeScript project as a graph of named nodes, not as a pile of files to edit directly. The graph source root is configured in `sail.config.json` under `graphSrc`. Each file under that configured directory defines exactly one public thing, and that thing is either a function, a type node, or a constant. The filename and exported symbol must match. `<graphSrc>/index.ts` is special: in a standalone sail project it is the async `main()` entrypoint, and in an embedded project it is the public surface that outside code should import from.

Because the codebase is constrained this way, graph operations are usually more useful than raw file edits. Start with `query` to find the right node, then use `graph` to inspect structure. With no id, `graph` shows the whole project graph. With an id, it shows the graph starting from that node. Use `read` only when you want the exact source for a chosen node. Use `patch` for surgical edits and `write` for full-file replacement. Use `sail test read|write|patch <id>` for the tests that belong to one node.

When acting as an agent, do not edit files in the configured graph source directory directly. Use sail commands so the graph stays valid, writes are checked, and all actions are logged. `write` replaces an existing node file or creates a missing one at `<graphSrc>/<id>.ts` or `<graphSrc>/<id>.tsx`, inferred from the source when needed, so always pass a full valid TypeScript file, not a fragment or arbitrary text. `patch` edits an existing node in place, either by exact find/replace or by unified diff from stdin. Tests live beside nodes as `<graphSrc>/<id>.spec.ts` or `<graphSrc>/<id>.spec.tsx`.

Only static local imports are allowed. Top-level executable code is not allowed except in standalone-project `<graphSrc>/index.ts` files that invoke `main()`. When sail is embedded into a larger TypeScript app, code outside the sail graph should import only from `<graphSrc>/index.ts`, not directly from internal sail node files. In Next.js projects, once an `app/**` file starts using sail, it should become a pure re-export stub from `<graphSrc>/index.ts` rather than containing local component logic. Run commands from the project root. Use `sail init` to bootstrap a project; it will also write `sail.config.json`. After function writes and patches, sail may warn when the node looks under-tested, when its tests fail, when outside code is importing sail internals directly, or when Next adapter files need to be refactored into pure re-export stubs. If a write opens test debt, import-boundary debt, or Next adapter debt, pay it back before the next implementation write: later node writes and patches will be blocked until the debt is cleared.

Every command is logged under `~/.sail`. Run `sail ui` to browse those logs locally.
